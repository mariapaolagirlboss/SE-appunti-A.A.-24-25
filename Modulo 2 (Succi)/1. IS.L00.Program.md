- **Modulo**: un gruppo logico di codici in relazione fra di loro, potrebbe essere un gruppo di classi in un linguaggio object-oriented oppure un gruppo di funzioni in un linguaggio strutturato o funzionale. 
	- Insieme di librerie di funzioni che eseguono compiti specifici (approccio procedurale)
	- Classi (object oriented programming)
		- **Modularità**: processo che raggruppa logicamente un insieme di codici in relazione fra di loro.
	![[Screenshot 2025-11-07 alle 13.37.26.png|215x102]]
- **Subsystem**: un raggruppamento logico di uno o più moduli. 
	- Esegue specifici compiti o una parte delle responsabilità (approccio procedurale)
	- Insieme di classi (object oriented programming)
			- esempio di subsystem: DBMS e error processing system
	![[Screenshot 2025-11-07 alle 13.39.38.png|214x146]]
- **Information hiding**: il modulo deve nascondere la sua implementazione interna. Il modulo è accessibile solo tramite interfaccia pubblica (?):
		- Niente accesso diretto ai dati interni e ai metodi privati
		- I dati sono accessibili solo attraverso un insieme ben definito di metodi di accesso
	- Utilizzo dell'astrazione per definire i moduli e le interfacce
	- Cambiare l'implementazione (senza cambiare l'interfaccia) non dovrebbe avere alcun effetto sul resto del sistema
- **Low coupling**: due moduli sono debolmente [[Glossario#Accoppiamento|accoppiati]] se
		- le interconnessioni e le dipendenze sono deboli
		- è più importante nascondere le informazioni che un accoppiamento elevato
	- Aumento dell'ordine dell'accoppiamento:
		- Metodi di un modulo che chiamano altri metodi di un altro modulo(??)
		- Data coupling/control coupling
		- Class di un modulo è la sottoclasse di un altro modulo
		- Moduli che fanno uso di specifiche features del compilatore o chiamale a specifiche procedure API del sistema operativo
		- I/O coupling
		- **EVITARE**: common coupling e content coupling
- **Modulo coeso**: tutti i suoi elementi sono indirizzati allo svolgimento di un singolo compito
	- L'aumento della grandezza della [[Glossario#Coesione|coesione]] può essere:
		- casuale: parti raggruppate senza motivo
		- logico: parti logicamente correlate, nessun'altra interazione
		- temporale: parti elaborate entro lo stesso limite di tempo
		- procedurale: il controllo passa da una parte all'altra
		- comunicativo: parti correlate dallo stesso I/O
		- sequenziale: output di una parte è input di un'altra
		- informativa: accesso alla stessa struttura di dati
		- funzionale: tutti gli elementi per un singolo concetto
- **Simplicity**: costruita soltanto per il codice che ne ha affettivamente bisogno
	- **Refactoring**: ristrutturazione di un sistema funzionante per renderlo più semplice
	- **Simplicity a livelli diversi**:
		- livello dei metodi: metodi brevi con piccole firme
		- livello dei moduli: una piccola interfaccia pubblica
		- livello del sistema:
			- evitare middle-man moduli e variabili globali
			- minimizza informazioni e control paths
			- mantiene gerarchie di ereditarietà piccole
	**N.B.** evita duplicazione di codici
